## 0. Read This First

Phase 2 shipped a working **flat-plane** 3D multiplayer game: two mirrored
houses rendered as extruded wall-boxes with flat tinted floors, a Blender
character with Idle/Walk, mouse-look third-person camera, full networked core
loop, host lobby, and AI bots. It plays — but it reads as **colored boxes**, not
places.

Phase 3 makes the world **look and feel like real houses** — roofs, windows,
furnished rooms, real materials and lighting — **without changing the server, the
movement model, the camera, or any gameplay rule.** This is a presentation and
environment-art pass on the existing flat plane. It is deliberately low-risk: the
multiplayer game keeps working exactly as it does today; it just stops looking
like a prototype.

**Build exactly what is specified here. Do not add features not listed. Do not
simplify. Stop and ask before making any architectural decision not covered
here.**

## 1. The One Architectural Decision (read before scoping)

**The world has no height axis.** The server model
(`server/src/zones.ts`, `getZoneAt(x, y)`) and the client controller are purely
2D on a flat plane — no gravity, no jump, no vertical floors. "Basement" and
"bedroom" are floor regions, not stacked levels.

Therefore **functional stairs and true multi-floor houses are OUT of Phase 3** —
they require a server height/floor axis and a real physics engine, which is a
later, deliberate phase (Master Plan Phase 5, after the Phase 4 physics work).
Building non-functional stairs now would imply verticality that does not exist
and mislead players.

Phase 3 delivers **everything that makes a house believable on a single floor**:
enclosing walls that already exist, **roofs that reveal the room you're in**,
**windows**, **furniture**, **materials**, and **lighting**. If you want true
verticality sooner, that is a scope change to raise now — otherwise we proceed
flat-plane. This keeps the working multiplayer stable and the cost low.

## 2. Scope

### 2.1 In scope
1. **Roofs with per-room reveal.** Give each house a roof so it reads as a
   building from outside, but the roof section above the room containing the
   **local player** hides (or fades) so the top-down-ish chase cam still sees
   inside — the standard "dollhouse" reveal. Reveal is per-room, driven by the
   local player's current zone (`getZoneAt`). Purely visual/local; never sent to
   the server, never affects other clients' logic.
2. **Windows.** Real window openings in exterior walls: frame + translucent glass
   material, sized and placed from the floor-plan data. **Cosmetic only** in
   Phase 3 (you cannot climb through them yet — that's Phase 5). They make
   exteriors read as houses and let a bit of light/vision through.
3. **Furniture & props per room**, generated by **headless Blender bpy scripts**
   and placed deterministically from a per-zone manifest:
   - Master bedrooms: bed, nightstand/dresser, a clear "cash stash" spot where
     bundles sit.
   - Living rooms: sofa, TV/console, coffee table, rug.
   - Basements: crates, shelving, pipes, a jail-cell corner (bars) where jailed
     players are held.
   - Backyards: fence line, a couple of plants/bushes, optional shed.
   - Garden (neutral middle): trees, bushes, a path, one central feature.
4. **Materials & lighting upgrade.** Replace flat vertex-colored slabs/walls with
   proper PBR-ish materials (baked textures are fine, generated via bpy or simple
   procedural), and upgrade `SceneManager` lighting: a directional "sun" with
   soft shadows + tuned ambient/hemisphere light, so interiors read as rooms with
   depth, not flat color fills. Use the `threejs-aaa-graphics-builder` skill for
   art direction, material/lighting choices, and render-budget discipline.
5. **A data-driven prop system** that places props per zone, marks which props
   are **solid** (added to `colliderRects`) vs **decorative** (no collision), and
   merges static geometry to stay within the render budget.

### 2.2 Explicitly OUT of scope
- Any change to `server/`, the schema, `zones.ts`, or the floor-plan rect data
  in `geometry/floorplan.ts` (you may READ these; do not change the numbers).
- True verticality, functional stairs, jump-through windows, gravity, jump
  (Phase 5).
- Character redesign, ragdolls, physics engine / Rapier (Phase 4).
- Audio, HUD/menu redesign, particles beyond simple ambient touches (Phase 6).
- Networking, deployment, Steam packaging (Phases 7–8).
- New gameplay mechanics, rules, zones, or doors.

Add a `world/` subtree; keep each file single-purpose. Do not bloat
`EnvironmentBuilder.ts` — refactor its responsibilities into the subtree if
needed, preserving its current output.

```
client/src/three/
├── SceneManager.ts            # extend: upgraded lighting + shadow setup
├── EnvironmentBuilder.ts      # existing shell/walls/floors (keep; may delegate)
├── world/
│   ├── RoofSystem.ts          # builds per-room roof panels; per-frame reveal by local zone

├── CameraRig.ts               # unchanged
├── CashBundleView.ts          # unchanged (bundles still placed by server state)
└── GameController.ts          # wire in RoofSystem reveal + pass prop colliders into the controller

assets/blender/
├── build_character.py         # existing
├── build_cashbundle.py        # existing
├── build_furniture.py         # NEW: beds, sofas, tables, crates, shelves, TV, etc. -> GLBs
├── build_nature.py            # NEW: trees, bushes, fence, plants -> GLBs
└── (generated GLBs go to client/public/models/props/)
```

## 5. Detailed Requirements

### 5.1 Roof system (`RoofSystem.ts`)
- Build a roof panel per interior room (bedroom/living/basement per house) plus a
  simple pitched or flat roof exterior read. Panels are keyed by `ZoneId`.
- Each frame, `GameController` passes the local player's current zone
  (`getZoneAt(controller.x, controller.z)`); `RoofSystem` hides/fades the panel
  over that room (and optionally adjacent rooms for readability) and shows the
  rest. Use a quick opacity fade, not a hard pop.
- Local-only and purely visual. Never networked; other players' roofs are driven
  by *their own* local zone on their own client.
- Backyards/garden are open-air (no roof).

### 5.2 Windows (`WindowBuilder.ts`)
- Add window openings to exterior house walls at data-defined spots (author a
  small `windows` table in `propManifest.ts`, in floor-plan coordinates). Each
  window = a frame (wood, matching `COLORS.doorFrame`) + a translucent glass quad.
- Windows are **cut into / overlaid on** the existing wall boxes visually; do
  **not** create passable gaps (no collision or zone change). Keep the merged
  static-geometry approach — windows join the walls mesh or a dedicated merged
  windows mesh.

### 5.3 Prop generation (bpy) & library (`PropLibrary.ts`, `build_*.py`)
- Author headless bpy scripts that output stylized low-poly GLBs to
  `client/public/models/props/`. Match the existing character/bundle art style
  (clean, readable, slightly exaggerated — this game is meant to look fun, not
  photoreal). Keep polycounts low.
- `PropLibrary` loads each GLB once and clones instances (mirror
  `CashBundleView`/`CharacterModel` load-once-clone pattern).
- Document each script's output at the top of the file (what meshes, rough
  dimensions in Blender units, target world scale).

### 5.4 Prop placement (`HouseDresser.ts`, `propManifest.ts`)
- `propManifest.ts` is plain data: for each zone, a list of
  `{ prop, x, y, rotation, solid }` entries in **floor-plan (pre-scale)
  coordinates**, scaled by `WORLD_SCALE` at load exactly like `floorplan.ts` and
  `zones.ts` do — keep raw numbers hand-comparable.
- `HouseDresser` instantiates props from `PropLibrary`, positions them (`x,y ->
  three x,z`), and returns the list of **solid** prop collider rects.
- `GameController` concatenates those solid rects into the array passed to
  `CharacterController` so players collide with beds/sofas/crates but walk past
  rugs/plants. Do **not** change the collision algorithm — just extend the rect
  list.
- Placement must be **deterministic** (no per-run randomness that desyncs what
  players see) and must not block doorways, the cash-stash spots, spawn points,
  jail-cell positions, or the threshold-crossing paths the core loop depends on.

### 5.5 Materials & lighting (`SceneManager.ts` + graphics skill)
- Upgrade lighting: one directional sun with soft shadow maps (reasonable
  resolution, not 4k), plus hemisphere/ambient fill so interiors aren't flat.
  Enable shadows on the renderer; set walls/props to cast, floors to receive
  (floors already `receiveShadow`).
- Move from pure vertex colors toward proper materials/textures where it clearly
  improves readability (floors per room type, walls, props). Baked textures from
  bpy or simple procedural materials are fine. Keep it stylized and cohesive.
- Respect a **render budget**: use the `threejs-debug-profiler` skill to keep
  draw calls and triangles in check (continue merging static geometry; instance
  repeated props). Target a stable 60fps on a mid laptop with 8 players in view.

### 5.6 Integration (`GameController.ts`)
- On `start`, after building the environment: build roofs, windows, and dress the
  houses; add all resulting meshes to the scene; feed solid prop rects into the
  local `CharacterController`; pass roof-relevant meshes to the camera's obstacle
  list only if they shouldn't be clipped through (roofs are hidden over the local
  room, so generally exclude them from camera raycast).
- Each frame, update the roof reveal from the local zone. Everything else in the
  tick loop stays byte-for-byte as it is.

## 6. Coordinate & Data Conventions (unchanged, restated)

- Server `(x, y)` == Three `(x, z)`; Three `y` is up; gameplay at `y ≈ 0`.
- All new placement data is written in **pre-scale floor-plan coordinates** and
  multiplied by `WORLD_SCALE` at load, matching `floorplan.ts`/`zones.ts`.
- `geometry/floorplan.ts` and `server/src/zones.ts` remain the single source of
  truth for zones/walls/doors. Read them; never fork or edit their numbers.

## 9. Definition of Done (Manual Verification — No Automated Tests)

Confirm by playing in browser tabs. This replaces, and must not become, test
code.

**Houses read as houses**
- [ ] From outside, each house looks like a building with a roof and windows —
      not open-topped colored boxes.
- [ ] Walking into a room hides/fades only that room's roof; you always see your
      own character and immediate room; the rest of the house keeps its roof.
- [ ] Windows show frames + glass in exterior walls and do not create walk-through
      gaps or change collision/zones.

**Rooms are furnished and still playable**
- [ ] Bedrooms, living rooms, basements, backyards, and the garden each have
      distinct, fitting props.
- [ ] Solid props (beds, sofas, crates) block movement; decorative props (rugs,
      plants) do not.
- [ ] No prop blocks a doorway, the cash-stash spots, spawn points, jail-cell
      positions, or the home-threshold deposit path. The full core loop
      (pickup / carry / deposit / lock / jail / rescue / steal-back) works
      exactly as before.

**Look & performance**
- [ ] Lighting + materials make interiors read with depth; the game looks like a
      real (stylized) place, not flat fills.
- [ ] Stable ~60fps with a full 4v4 + bots match in view (profiled via the
      debug-profiler skill, observed manually — no committed perf test).

**No regressions / structure**
- [ ] Server, schema, zones, floor-plan numbers, camera, HUD, and all network
      messages are unchanged.
- [ ] New code lives in `three/world/` + `assets/blender/`, each file
      single-purpose; no test files exist; no dead code.
